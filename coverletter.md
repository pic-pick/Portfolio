## 분석 대상 텍스트 원문

### Hero 섹션

**뱃지**: `Frontend Developer Portfolio`

**타이틀 (H1)**:
> 사용자의 경험을
> 더 자연스럽게 만드는 개발자
> 오승환입니다.

**본문 1단락**:
> 프론트엔드 개발자로서 작은 디테일이 사용자 경험을 만든다고 믿습니다. 화면의 섬세한 요소와 직관적인 상호작용을 통해 사람들이 자연스럽게 서비스를 이용할 수 있도록 고민하며 개발합니다.

**본문 2단락**:
> 사용자의 특별한 경험을 기술로 연결하고 더 나은 서비스로 이어질 수 있도록 문제를 고민하고 해결합니다. 사용자 피드백과 테스트를 통해 발전하는 서비스를 만드는 것이 저의 목표입니다.

---

### About 섹션

**타이틀 (H2)**:
> 배우고 만들며 성장하는
> 프론트엔드 개발자

**본문 1단락**:
> 사용자 경험을 중심으로 문제를 해결하는 프론트엔드 개발자입니다. React와 TypeScript, JavaScript 기반으로 인터페이스를 구현하며 REST API 연동과 상태 관리를 통해 실제 서비스 흐름을 이해하는 개발 경험을 쌓아왔습니다.

**본문 2단락**:
> 단순히 화면을 만드는 것에 그치지 않고, 사용자가 어떤 흐름으로 서비스를 이용하는지 고민하며 더 직관적인 UI와 안정적인 구조를 만드는 것을 중요하게 생각합니다. 이 포트폴리오는 제가 경험한 프로젝트와 기술, 그리고 앞으로 어떤 개발자로 성장하고 싶은지를 보여주기 위해 구성했습니다.

## 1. 전체 구조 흐름 평가

포트폴리오의 소개 텍스트 구조는 이렇게 되어 있다:

```
Hero 타이틀  →  Hero 본문 1  →  Hero 본문 2
     ↓
About 타이틀  →  About 본문 1  →  About 본문 2
```

**이상적인 흐름**은 각 단계가 이전 단계를 **심화하거나 다른 각도로 보완**하는 것인데, 현재 구조는 Hero에서 말한 내용을 About이 거의 같은 내용을 다른 말로 반복하는 구조에 가깝다.

| 위치 | 현재 메시지 | 이상적 역할 |
|---|---|---|
| Hero 타이틀 | "사용자 경험을 자연스럽게 만드는 개발자" | ✅ 핵심 정체성 선언 |
| Hero 본문 1 | "작은 디테일이 사용자 경험을 만든다 믿음" | ⚠️ 타이틀과 너무 유사 |
| Hero 본문 2 | "사용자 경험 기술로 연결, 피드백 통해 발전" | ⚠️ 추상적, 방향성 불명확 |
| About 타이틀 | "배우고 만들며 성장하는 개발자" | ⚠️ 너무 generic |
| About 본문 1 | "사용자 경험 중심, React/TS/JS 사용" | ❌ Hero 타이틀과 동일 메시지 반복 |
| About 본문 2 | "직관적 UI 고민, 포트폴리오 소개" | ❌ 자기소개서 안에서 자기소개서를 설명 |

## 2. 문제점 상세 분석

### 2-1. "사용자 경험" 키워드 과잉 반복

전체 소개 텍스트에서 "사용자 경험" 또는 "사용자" 관련 표현이 **5회** 등장한다.

| 위치 | 표현 |
|---|---|
| Hero 타이틀 | "사용자의 경험을 더 자연스럽게 만드는" |
| Hero 본문 1 | "작은 디테일이 **사용자 경험**을 만든다" |
| Hero 본문 2 | "**사용자의** 특별한 경험을 기술로 연결" |
| About 본문 1 | "**사용자 경험**을 중심으로 문제를 해결" |
| About 본문 2 | "**사용자가** 어떤 흐름으로 서비스를 이용" |

첫 타이틀에서 강하게 선언한 메시지가 뒤에서 계속 반복되면서 점점 희석된다. 타이틀에서 "사용자 경험"으로 임팩트를 준 뒤, 본문에서는 **다른 축**의 강점(기술적 역량, 팀 협업, 문제 해결 방식 등)으로 전환해야 메시지가 쌓인다.

---

### 2-2. "고민" 반복 — 소극적 인상

"고민"이라는 단어가 3회 반복된다.

- Hero 본문 1: "**고민**하며 개발합니다"
- Hero 본문 2: "문제를 **고민**하고 해결합니다"
- About 본문 2: "**고민**하며 더 직관적인 UI"

"고민"은 성찰적인 개발자 이미지를 주려는 의도로 보이지만, 반복되면 "아직 확신이 없는 개발자"처럼 읽힌다. 특히 채용자 관점에서는 "고민한다"보다 "분석하고 설계한다", "개선한다"처럼 능동적인 동사가 훨씬 강한 인상을 준다.

---

### 2-3. Hero 2단락 — 주어와 연결 고리가 불명확

> "사용자의 특별한 경험을 기술로 연결하고 더 나은 서비스로 이어질 수 있도록 문제를 고민하고 해결합니다."

이 문장은 두 개의 동작이 묶여 있는데 주어가 모호하다.

- "기술로 연결하고" → 내가 연결한다는 건가, 기술이 연결한다는 건가?
- "더 나은 서비스로 이어질 수 있도록" → 주어 없이 수동적

또한 1단락과 2단락의 연결이 끊긴다:
- 1단락: "작은 디테일이 중요하다 → 자연스럽게 이용할 수 있도록 개발"
- 2단락: "특별한 경험을 기술로 연결 → 피드백과 테스트로 발전"

두 단락이 서로 다른 이야기를 하고 있어서, Hero 본문이 하나의 통일된 메시지를 전달하지 못한다.

---

### 2-4. Hero 2단락 마지막 문장 — "목표"로 끝나는 구조 문제

> "사용자 피드백과 테스트를 통해 발전하는 서비스를 만드는 것이 저의 목표입니다."

첫 만남에서 "저의 목표입니다"로 끝나는 건 어색하다. Hero는 방문자가 이 사람이 누구인지 파악하는 첫 화면인데, 목표를 선언하면 아직 그 목표를 이루지 못했다는 인상을 줄 수 있다. "이미 하고 있는 것"으로 서술하는 편이 더 설득력 있다.

---

### 2-5. About 타이틀 — 차별화 없는 표현

> "배우고 만들며 성장하는 프론트엔드 개발자"

이 표현은 거의 모든 신입/주니어 개발자에게 붙일 수 있는 문구다. 오승환이라는 사람의 고유한 방향성이나 강점이 전혀 드러나지 않는다. About 타이틀은 Hero 타이틀과 다른 각도에서 이 사람을 설명해야 한다.

예를 들어:
- 사용자 경험을 위해 백엔드까지 이해하는 프론트엔드 개발자
- 서비스 흐름 전체를 설계하는 프론트엔드 개발자
- AI와 금융을 배경으로 하는 프론트엔드 개발자

→ About 타이틀에서 Hero에서 선언한 "사용자 경험"을 뒤받침하는 **구체적인 배경**을 드러내는 것이 이상적이다.

---

### 2-6. About 본문 1단락 — Hero와 메시지 중복

> "사용자 경험을 중심으로 문제를 해결하는 프론트엔드 개발자입니다."

Hero 타이틀과 거의 동일한 자기소개를 About에서 또 하고 있다. 방문자가 스크롤을 내려서 About을 읽는 순간, 이미 Hero에서 본 내용이 반복된다.

About 본문은 Hero에서 선언한 정체성을 **증명하는 구간**이어야 한다. "사용자 경험을 중심으로 문제를 해결한다"고 했다면, About에서는 "실제로 어떤 방식으로 그걸 해왔는지"가 나와야 한다.

---

### 2-7. About 본문 1단락 후반 — "이해하는 개발 경험을 쌓아왔습니다"

> "REST API 연동과 상태 관리를 통해 실제 서비스 흐름을 이해하는 개발 경험을 쌓아왔습니다."

"이해하는 경험을 쌓아왔습니다"는 자신감이 없는 표현이다. 실제로 Re:view, 댕댕트립, 차트인사이트를 완성한 개발자인데, 이 표현은 마치 아직 배우는 중인 것처럼 들린다. "구현해왔습니다", "개발해왔습니다" 정도로도 훨씬 강하다.

---

### 2-8. About 본문 2단락 마지막 문장 — 포트폴리오 안에서 포트폴리오 설명

> "이 포트폴리오는 제가 경험한 프로젝트와 기술, 그리고 앞으로 어떤 개발자로 성장하고 싶은지를 보여주기 위해 구성했습니다."

방문자는 이미 포트폴리오를 보고 있는 상황에서, 포트폴리오가 무엇인지 설명하는 문장이 About 자기소개의 마지막을 차지하고 있다. 이 문장은 삭제하거나 다른 내용으로 대체해야 한다. 자기소개의 마지막은 "앞으로 어떤 개발자로 함께하고 싶다"는 방향성이나 초대로 마무리되어야 인상이 남는다.

---

## 3. 긍정적 요소

- Hero 타이틀 "사용자의 경험을 더 자연스럽게 만드는 개발자" — 명확하고 기억에 남는 선언이다
- Hero 본문 1단락의 "작은 디테일이 사용자 경험을 만든다" — 개발 철학이 느껴지는 좋은 문장
- About 본문 2단락의 "단순히 화면을 만드는 것에 그치지 않고" — 백엔드까지 이해하는 풀스택적 사고를 암시하는 좋은 도입부
- 기술 스택(React, TypeScript, Spring Boot, REST API)을 Hero 하단 뱃지로 보여주는 구성 — 텍스트 설명 없이 역량을 시각적으로 전달

## 4. 개선 방향 제안

### 원칙

| 섹션 | 역할 | 핵심 메시지 방향 |
|---|---|---|
| Hero 타이틀 | 정체성 선언 (1줄) | 오승환이 누구인지 한 방에 |
| Hero 본문 | 개발 철학 + 방법론 | "왜 이 방식으로 개발하는가" |
| About 타이틀 | 타이틀 보완 | Hero와 다른 각도의 강점 |
| About 본문 | 역량 증명 + 지향점 | "어떤 배경과 경험으로 성장했는가" |

---

### Hero 본문 개선 방향

**현재 문제**: 두 단락이 모두 "사용자 경험 고민"으로 수렴함.

**제안**:
- 1단락: 개발 철학 (왜 프론트엔드인가, 무엇을 중요하게 생각하는가)
- 2단락: 실천 방법 (어떻게 구현하는가, 팀/서비스에서 어떤 역할을 하는가)

예시 방향:
```
[1단락]
작은 인터랙션 하나가 서비스에 대한 신뢰를 만든다고 생각합니다.
화면의 반응 속도, 버튼의 피드백, 상태 전환의 자연스러움—
사용자가 의식하지 못할 만큼 자연스러운 경험을 만드는 것이
저의 개발 기준입니다.

[2단락]
React와 TypeScript로 인터페이스를 구현하고,
Spring Boot 기반 REST API와 연동하며 서비스 전체 흐름을
함께 설계해왔습니다. 팀 프로젝트와 개인 프로젝트를 통해
기획부터 배포까지 직접 경험하며 개발자로서의 관점을 넓혀왔습니다.
```

---

### About 타이틀 개선 방향

**현재**: "배우고 만들며 성장하는 프론트엔드 개발자"

**제안**: Hero가 "사용자 경험"을 선언했다면, About은 그걸 뒷받침하는 배경을 드러내야 한다.

```
예시 A: "프론트엔드와 백엔드를 함께 이해하는 개발자"
예시 B: "서비스 전체 흐름을 설계하는 프론트엔드 개발자"
예시 C: "AI와 금융 기반 위에 프론트엔드를 쌓아온 개발자"
```

**예시 B 방향 추가 5개** — "서비스 흐름 / 구조 / 맥락을 함께 보는 프론트엔드 개발자" 주제:

[//]: # (G로 적용하자)
```
예시 D: "화면 너머 서비스 구조까지 함께 보는 프론트엔드 개발자"
예시 E: "기획에서 배포까지 흐름을 이해하며 만드는 프론트엔드 개발자"
예시 F: "프론트엔드로 시작해 서비스 전체를 설계하는 개발자"
예시 G: "UI 너머 사용자 여정 전체를 설계하는 프론트엔드 개발자"
예시 H: "백엔드와 대화하며 서비스를 완성하는 프론트엔드 개발자"
```

> **추천**: 예시 G — "사용자 여정"이라는 표현이 Hero 타이틀의 "사용자 경험"과 연결되면서도 "설계"라는 능동적 역할을 더해 차별화됨. 단, 한 줄에 너무 길면 두 줄로 분리 (예: "UI 너머 / 사용자 여정 전체를 설계하는 프론트엔드 개발자")

---

### About 본문 개선 방향

**현재 문제**: Hero 타이틀과 동일한 자기소개 반복, 자신감 없는 표현, 포트폴리오 안에서 포트폴리오 설명.

**제안 방향**:
- 1단락: 나의 개발 경험 배경 (어떤 과정으로 이 자리에 왔는가)
- 2단락: 나의 개발 방식과 지향점 (어떤 개발자가 되고 싶은가)


```


[1단락]

인공지능융합학과에서 알고리즘과 데이터 기반 사고를 배우고,
멀티캠퍼스 풀스택 아카데미를 거쳐 Java부터 Spring Boot,
React, TypeScript까지 실무 중심으로 익혔습니다.
팀 프로젝트와 개인 프로젝트를 통해 기획부터 배포까지
직접 경험하며 서비스 전체 흐름을 이해하는 개발자로 성장해왔습니다.

[2단락]
저는 기능이 동작하는 것을 넘어, 사용자가 불편함을 느끼지 않는
인터페이스를 만들고 싶습니다. 팀 안에서는 백엔드와 적극적으로
소통하며 서비스 흐름을 함께 설계하고, 작은 디테일도 놓치지 않는
프론트엔드 개발자로 성장하고 있습니다.
```

## 5. 우선순위별 수정 권장 사항

| 우선순위 | 항목 | 이유 |
|---|---|---|
| 🔴 높음 | About 본문 마지막 문장 삭제 ("이 포트폴리오는~") | 방문자 관점에서 맥락 없는 메타 설명 |
| 🔴 높음 | About 본문 1단락 앞 문장 수정 ("사용자 경험을 중심으로~") | Hero 타이틀과 동일 메시지 반복 |
| 🟠 중간 | Hero 2단락 "목표입니다" 표현 → "하고 있습니다"로 전환 | 이미 실천 중인 것으로 서술해야 강함 |
| 🟠 중간 | "고민" 3회 반복 → 능동적 동사로 교체 | 소극적 인상 제거 |
| 🟠 중간 | About 타이틀 구체화 | 오승환만의 배경이 드러나도록 |
| 🟡 낮음 | Hero 1단락과 타이틀 메시지 차별화 | 메시지 심화를 위해 |
| 🟡 낮음 | "이해하는 개발 경험을 쌓아왔습니다" → "구현해왔습니다" | 자신감 있는 표현 |

---

## 6. 요약

Hero와 About 소개 텍스트는 **"사용자 경험"이라는 하나의 키워드에 너무 많이 의존**하고 있으며, 두 섹션이 서로 심화하지 못하고 반복된다. 핵심 키워드를 Hero에서 강하게 선언하고, About에서는 그것을 가능하게 하는 배경과 역량으로 증명하는 구조로 재편하면 전체 흐름이 훨씬 설득력 있어진다.

또한 "고민", "이해하는 경험을 쌓아왔습니다", "목표입니다" 같은 소극적 표현들을 능동적이고 자신감 있는 서술로 바꾸는 것이 필요하다. 실제로 여러 프로젝트를 완성하고 수상까지 한 개발자의 소개글로서 현재 표현은 그 경험을 충분히 반영하지 못하고 있다.
